# 🚀Kotlin Study
코틀린 개념을 단 2회차 만에 압축해서 내 것으로 만드는 스터디입니다. 각자 공부한 내용을 이곳에 기록하고 세미나를 통해 공유합니다.

## 📌 1. 브랜치(Branch) 생성 규칙
작업을 시작하기 전, 항상 `main` 브랜치 최신화 후 **자신의 이름이 들어간 회차 브랜치**를 생성합니다.
* **형식:** `session회차/이름`
* **예시:** `session1/subin`, `session1/jiyoung`


## 📌 2. 폴더 및 파일 구조

충돌(Merge Conflict)을 방지하기 위해, 파일은 회차 폴더 생성 후 그 안에 **반드시 본인의 이름으로 된 폴더**를 생성해야 합니다. 최상단(루트)에 바로 파일을 만들면 PR을 합칠 때 다른 팀원과 충돌이 발생합니다!

* **형식:** `session회차/본인이름/공부한내용.md`
* **올바른 예시:** `session1/subin/1_Variables.md`
* **잘못된 예시:** `1_Variables.md` (최상단에 그냥 만들면 안 됨!)


**[작업 순서]**

1. 본인 브랜치로 이동한 상태에서
2. 프로젝트 최상단에 `session1` 폴더를 직접 만듭니다.
3. 그 안에 `(본인 이름)` 폴더를 만듭니다.
4. 해당 폴더 안에 마크다운(`.md`) 파일이나 예제 코드 파일 등을 넣고 작성합니다.


## 📌 3. 커밋 메시지 (Commit Message) 규칙

스터디 저장소의 특성을 살려 직관적이고 깔끔하게 작성합니다. 영어 태그 뒤에 콜론(`:`)을 붙이고 띄어쓰기를 한 번 합니다.

* **형식:** `태그: [session회차] 작업 내용`

| 태그 | 설명 | 커밋 메시지 예시 |
| --- | --- | --- |
| `docs` | 마크다운 문서 작성 및 수정 (가장 많이 씀) | `docs: [session1] 수빈 1주차 코틀린 기초 정리` |
| `feat` | 코틀린 예제 코드나 샘플 프로젝트 추가 | `feat: [session2] 지영 스코프 함수 테스트 코드 추가` |
| `fix` | 문서 내용 수정 및 오타 수정 | `fix: [session1] 상속 개념 설명 오타 수정` |
| `chore` | 폴더 생성, 파일 이동 등 자잘한 작업 | `chore: [session1] 폴더 구조 세팅` |


## 📌 4. PR (Pull Request) 및 Merge 규칙

1. 깃허브 홈페이지에서 `main` 브랜치를 향해 PR을 생성합니다.
2. PR 제목은 커밋 메시지와 동일하게 적어주세요. (예: `[session1] subin 1주차 정리 완료`)
3. 세미나(스터디 모임)가 끝난 후, 각자의 PR을 `main` 브랜치로 **Merge** 합니다.
